1 # FFmpeg Encryption
2 
3 This document describes how the encryption info works in FFmpeg. This is for
4 future Shaka Embedded developers who want to add support for different kinds
5 of encryption info to FFmpeg in the future. See
6 [libavutil/encryption_info.h][1].
7 
8 [1]: https://github.com/FFmpeg/FFmpeg/blob/master/libavutil/encryption_info.h
9 
10 There are two kinds of encryption info: (1) frame-specific encryption info
11 stored as `AVEncryptionInfo`, and (2) encryption initialization info (e.g.
12 `pssh` data) stored as `AVEncryptionInitInfo` on the `AVStream`.
13 
14 Also note, in FFmpeg, the encoded (demuxed) frames are called packets
15 (`AVPacket`), while the decoded frames are called frames (`AVFrame`). This will
16 still use "frame" to refer to an `AVPacket`.
17 
18 
19 ## Side data
20 
21 The encryption info is stored as side data on either an `AVPacket` or an
22 `AVStream`. The side-data itself is binary data. `encryption_info.h` defines
23 types and methods that can be used independent of container formats. You need
24 to use those methods to convert between the binary format and the generic
25 object.
26 
27 If you add a side-data to an object that already has a side-data of that type,
28 the old value is replaced. So if there are multiple values (e.g. for multiple
29 `pssh` support), you'll need to get the old value, parse it, and combine it with
30 the new info you are adding.
31

33 ## Frame-specific encryption info
34 
35 First, you'll need to update the demuxer to parse the encryption info. This
36 will likely involve editing the internal state to include this info. It would
37 be convenient if you used an `AVEncryptionInfo` object now, but you don't have
38 to.
39 
40 Next, you need to find where the demuxer creates the frames (`AVPacket`). This
41 is usually named `*_read_packet`. In this method, before the frame is returned
42 to the caller, you need to add the side-data to it. If you haven't already,
43 create an `AVEncryptionInfo` object and fill it with the required info. Then
44 do something like:
45 
46 ```c
47 // Convert the object to a serialized binary format.
48 size_t side_data_size;
49 uint8_t *side_data =
50  av_encryption_info_add_side_data(encryption_info, &side_data_size);
51 if (!side_data)
52  return AVERROR(ENOMEM);
53 
54 // Add the data to the packet; this takes ownership of |side_data|.
55 int ret = av_packet_add_side_data(
56  &pkt, AV_PKT_DATA_ENCRYPTION_INFO, side_data, side_data_size);
57 if (ret < 0) {
58  av_free(side_data);
59  return ret;
60 }
61 // Remember to free |encryption_info|.
62 ```

65 ## Encryption initialization info
66 
67 Like before, you'll need to update the demuxer to parse whatever initialization
68 info is required. It should be put into a new `AVEncryptionInitInfo` struct.
69 Some demuxers support multiple initialization infos, so before setting the
70 stream's side-data, you'll need to determine if there is any existing side-data.
71 
72 ```c
73 // Find any existing side-data.
74 size_t old_side_data_size;
75 uint8_t *old_side_data = av_stream_get_side_data(
76  stream, AV_PKT_DATA_ENCRYPTION_INIT_INFO, &old_side_data_size);
77 if (old_side_data) {
78  // Convert the side-data to a struct and add it to the liked list.
79  AVEncryptionInitInfo *old_init_info = av_encryption_init_info_get_side_data(
80  old_side_data, old_side_data_size);
81  if (old_init_info) {
82  init_info->next = old_init_info;
83  } else {
84  return AVERROR(ENOMEM);
85  }
86 }
87 
88 // Convert the object to a serialized binary format.
89 size_t side_data_size;
90 uint8_t *side_data = av_encryption_init_info_add_side_data(
91  init_info, &side_data_size);
92 if (!side_data)
93  return AVERROR(ENOMEM);
94 
95 // Add the data to the stream; this takes ownership of |side_data|.
96 int ret = av_stream_add_side_data(
97  stream, AV_PKT_DATA_ENCRYPTION_INIT_INFO, side_data, side_data_size);
98 if (ret < 0) {
99  av_free(side_data);
100  return ret;
101 }
102 // Remember to free |init_info|. Note that av_encryption_init_info_free will
103 // free child members.
104 ```
