1 # ABI Guidelines
2 
3 This document describes guidelines for maintaining ABI compatibility. As a C++
4 project, we need to consider an app that loads a newer version of the library
5 while they compiled using an older version of the headers. ABI
6 (application-binary interface) specifies how the executable will interact with
7 the library at a binary level. By maintaining ABI compatibility, an app can
8 load newer versions of the library using a `.dll` or `.so` file and it will
9 still work.
10 
11 We only maintain ABI compatibility within major release numbers. v1.1 will be
12 compatible with v1.4, but not compatible with v2.1. We make no promises about
13 partial compatibility between major versions. It should be assumed that v2 is
14 completely incompatible with v1.
15 
16 We also only maintain ABI compatibility with release versions and release
17 branches; master branch can break ABI at any time. Any commit that will break
18 ABI should have a line in the commit message with `Abi-Breaking:` and a reason.

21 ## Removing Features
22 
23 When something is to be removed from the library, it can only be done on a major
24 version release. Once a feature is decided to be removed, the next minor
25 release should mark the feature as deprecated so the compiler will warn the app
26 and a log will be printed. We will make at least one minor release before
27 removing a feature.
28 
29 For example, if we wanted to remove a feature, we would mark it deprecated in
30 v1.4. Then when we release v2 we will remove the feature. We will not remove
31 any features in v2 unless they were marked as deprecated in the previous minor
32 release.

35 ## Guidelines for Developers
36 
37 These guidelines are based on the guidelines by the KDE team:
38 https://community.kde.org/Policies/Binary_Compatibility_Issues_With_C%2B%2B
39 
40 We only need to worry about ABI on the public types. This means that we are
41 completely free to change anything about internal types. This includes
42 implementations of public types. For example, ImplementationHelper is a public
43 interface type, so we need to think about ABI for it; but the type that
44 implements that interface can be changed without breaking public ABI.
45 
46 
47 ### What you can change
48 
49 - Anything in a `.cc` file, `.js` file, or an internal-only `.h` file.
50 - Add new static data members.
51 - Add new non-virtual functions (must appear last).
52 - Add new virtual functions (must appear last) (may not work on Windows).
53 - Add a new constructor.
54 - Change a private Impl class definition.
55 - Add values to an `enum` or `enum class`.
56  - You can't change existing values.
57  - You cannot change the underlying type (e.g. from `uint8_t` to `uint16_t`).
58 - Rename private members.
59 
60 ### What you cannot change
61 
62 - The signature of a public function (return type, arguments, `const`).
63  - You cannot add an argument even if it has a default value.
64 - The accessibility of an existing member.
65  - You cannot even make a private member public.
66 - Class hierarchy.
67 - Add new virtual methods if the app will inherit from it.
68 - Add new non-static data members.
69 - Change the order of members.
70 - Insert a member before an existing member.
71 
72 ### Adding new types
73 
74 If you define a new public type, you should do the following:
75 
76 - Include a virtual, non-inline destructor.
77  - Can be non-virtual if type is `final` or the type shouldn't be destroyed by
78  the app.
79 - Include a non-inline copy and move constructor/assignment operator.
80  - If the type shouldn't be moved/copied, explicitly delete them.
81 - Include an Impl type and private member, even if it doesn't use it.
82  - If the type is an interface, you don't need this.
83 - Mark type as `final` unless it is an interface.
84 - `enum` or `enum class` should have an explicit base type.
